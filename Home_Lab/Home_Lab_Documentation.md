## Overview & Mission Statement

This lab simulates an internal network intrusion in a controlled home-lab environment.

It demonstrates how an attacker can perform automated reconnaissance and SMB credential attacks, while a Blue Team monitoring node observes all traffic using a Physical Ethernet Bridge on the HP ZBook (Detector).

By bypassing typical Layer 2 switch limitations, the Detector achieves full packet visibility, enabling real-time detection of attacks that would normally remain invisible on a standard network.

---

## Lab Objectives

---

The objectives of this lab are to:

---

- **Demonstrate Offensive Automation**

  - Use custom Python tooling (`smb_brute.py`) together with common penetration-testing tools such as **Hydra** and **Metasploit**.
  - Perform automated **SMB credential brute-force attacks** and basic network reconnaissance against the target system.

---

- **Deploy a Network Monitoring Node**

  - Run a Python-based intrusion detection script (`ids_monitor.py`) on the **HP ZBook (Detector)**.
  - Monitor network traffic in real time and identify suspicious activity targeting the victim machine.

---

- **Create a Transparent Monitoring Point**

  - Configure the HP ZBook as a **physical Ethernet bridge** between the network and the victim system.
  - Allow packet inspection of traffic that would normally remain hidden on a switched **Layer 2 network**.

---

- **Correlate Multi-Source Evidence**

  - Compare alerts generated by the Python IDS with **Windows Security Event ID 4625 (Failed Logon)** on the victim system.
  - Use this correlation to confirm brute-force activity with higher confidence.

---

- **Evaluate Defensive Improvements**

  - Analyze the observed attack behavior after the simulation.
  - Propose defensive improvements such as:
    - Account lockout policies
    - Firewall rule adjustments
    - Improved monitoring and alerting

---

## Attack & Detection Simulation
---

This section provides an overview of the attack simulation and monitoring setup.  
It explains **what happens during each phase**, without including live results or screenshots.

---

### Attack Overview

The attack follows a typical intrusion chain:

1. **Reconnaissance** – Scanning the victim system to identify open ports and running services.
2. **Service Identification** – Determining that the SMB service is exposed on TCP port 445.
3. **Credential Brute Force** – Using automated tools to identify valid login credentials.
4. **SMB Authentication** – Logging in with the discovered credentials to access shared resources.
5. **File Access** – Navigating shared directories and retrieving files to simulate data exfiltration.

---

### Monitoring Overview

To observe the attack traffic:

- A **transparent Ethernet bridge** is created using the HP ZBook between the attacker and victim systems.
- A Python-based IDS (`ids_monitor.py`) runs on the HP ZBook, analyzing network traffic for repeated SMB authentication attempts and bursts of suspicious activity.
- Alerts are generated in real time without interfering with the attack flow.

---

This section **explains the simulation workflow**.  
Actual commands, packet captures, and IDS outputs are included in the dedicated **“Attack Execution and Results”** section.

# Layer 2 Network Bridge Configuration

---

### Step 1: Network Adapter Reset (Detector Node)

To ensure a clean configuration, all previous routing and Network Address Translation (NAT) rules were disabled on the **HP ZBook**.

- Navigated to **Windows Network Connections** (`ncpa.cpl`)
- Accessed **Properties → Sharing** tab for both the **Wi-Fi** and **Ethernet** adapters
- Verified that **Internet Connection Sharing (ICS)** was completely **unchecked**
- Set both adapters to **Obtain an IP address automatically (DHCP)** in their **IPv4 settings**

---

### Step 2: Creating the Layer 2 Bridge (Detector Node)

The **Wi-Fi interface** and the **physical Ethernet interface** on the **HP ZBook** were fused to create a seamless **Layer 2 conduit**.

- In the **Network Connections** window, highlighted both the **Wi-Fi** and **Ethernet** adapters simultaneously
- Right-clicked the selection and executed **Bridge Connections**

Windows generated a new virtual adapter named **Network Bridge**, successfully linking the wireless network to the physical Ethernet cable.

# Python IDS Script Documentation

---

The goal of this phase was to successfully implement a Layer 2 IDS (Intrusion Detection System) that remains invisible to both the attacker and the victim. By leveraging a hardware bridge configuration on a Windows-based detector (HP ZBook), I successfully intercepted and analyzed malicious traffic without altering the network’s logical topology.

---

## Netowrk Topology

[ Attacker: Dell ] <--- Wi-Fi ---> [ Detector: HP ZBook (Bridge) ] <--- Ethernet ---> [ Victim: Toshiba ]
       |                                     |                                         |
 (Kali Linux/Nmap)                    (Python IDS Script)

---

---

### Script Components

#### Target Definition


```python
VICTIM_IP = "192.168.1.xx"
```


- Defines the internal IP address of the **Toshiba victim system** that the IDS is configured to protect.

---

#### Protocol Filtering

```python
if packet.haslayer(IP) and packet.haslayer(TCP):
```

- Filters captured packets to inspect only **IP and TCP traffic**, which are commonly used during reconnaissance and connection attempts.

---

#### Destination Verification

```python
if packet[IP].dst == VICTIM_IP:
```

- Ensures the IDS analyzes only traffic **destined for the protected host**.

---

#### Alert Generation

```python
print(f"[CRITICAL ALERT] Inbound attack detected!")
```

- When suspicious traffic is identified, the script generates a **high-priority alert** in the terminal.

---

#### Sniffer Engine

```python
sniff(prn=detect_attack, store=0)
```

- Continuously monitors **live network traffic** passing through the bridge.
- `store=0` prevents packets from being stored in memory during high-traffic events.
